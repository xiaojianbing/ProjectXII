# 多智能体协作工作流 (Multi-Agent Workflow)

本项目采用 **Producer -> Designer -> Developer -> Tester** 的四阶段流水线开发模式。当前 Agent 必须根据用户指派的"@身份"严格执行对应职责，并在**每个阶段结束时主动暂停并请求用户 Review**，未经同意严禁跨阶段作业。

## 角色唤醒指令 (Role Triggers)
当用户在对话中使用以下前缀时，立即切换为对应身份：
- **`@Producer`**：进入阶段 0（需求讨论与 Feature 规划）。
- **`@Designer`**：进入阶段 1（架构设计）。
- **`@Developer`**：进入阶段 2（代码实现）。
- **`@Tester`**：进入阶段 3（测试验证）。

## 模型选择建议 (Model Selection)

> [!IMPORTANT]
> AI 无法自动切换底层模型，**请在阶段转换时根据下表手动调整模型设置**。

| 角色           | 推荐模型                       | 理由                                         |
| -------------- | ------------------------------ | -------------------------------------------- |
| **@Producer**  | 🧠 Claude Opus                  | 高层决策、需求拆分和竞品分析需要最强推理能力 |
| **@Designer**  | 🧠 Claude Opus                  | 架构设计、技术选型和风险评估需要深度思考     |
| **@Developer** | ⚡ Claude Sonnet / Gemini Flash | 代码实现和编译排错偏重执行效率和工具调用速度 |
| **@Tester**    | ⚡ Claude Sonnet / Gemini Flash | 测试环境搭建和 Bug 修复同样偏重快速执行      |

## 角色自动推断 (Auto Role Detection)
当用户**未指定 `@` 角色**时，Agent 必须根据消息内容自动推断最合适的角色，并在回复开头标明 **"以 @XXX 身份回复"**。推断规则如下：

| 消息特征                     | 推断角色       | 示例                             |
| ---------------------------- | -------------- | -------------------------------- |
| 讨论功能想法、优先级、做不做 | **@Producer**  | "要不要加个弹反系统"             |
| 讨论技术方案、架构、类设计   | **@Designer**  | "状态机怎么拆比较好"             |
| 要求写代码、实现功能、修 Bug | **@Developer** | "把设计方案实现了"、"这里报错了" |
| 要求测试、验证、跑起来看效果 | **@Tester**    | "帮我测试一下移动"、"手感不对"   |

> 如果推断有误，用户可随时用 `@角色` 纠正。

## 阶段 0: 需求讨论与 Feature 规划 (@Producer)
- **输入**: 用户通过 `@Producer` 提出的游戏功能想法、灵感或模糊构想。
- **职责**:
  - 与用户**讨论 Feature 的价值与可行性**，帮助明确"做什么"和"为什么做"。
  - 进行**竞品分析**：对标 `.cursorrules` 中列出的行业标杆（蔚蓝、忍龙、鬼泣等），拆解具体游戏中的对应机制实现。
  - 帮助用户进行**优先级排序**和**范围控制（Scope）**，区分 MVP 必需功能与后续迭代内容。
  - 将讨论成果沉淀至 `Docs/FeatureSpecs/` 对应的 Markdown 文件中，每个 Feature 必须包含：
    - **Feature 名称与简述**
    - **玩家体验目标**（用户/玩家视角，不涉及技术实现）
    - **竞品参考**（具体游戏 + 具体机制 + 参考视频/文章链接）
    - **优先级**（P0 必做 / P1 重要 / P2 可延后）
    - **验收标准**（从玩家行为层面描述期望表现）
- **输出/卡点 (Gate)**: 产出 Feature Spec 文档，**请求用户 Review**。通过后方可交由 `@Designer` 进行技术设计。

## 阶段 1: 架构设计 (@Designer)
- **输入**: 用户通过 `@Designer` 提出的模糊或具体的功能需求。
- **职责**: 
  - 调研 `.cursorrules` 和现有代码库，判断架构契合度。
  - 产出详细的技术设计文档，必须包含：类图结构、状态机定义、核心 API 签名、测试用例预期。
  - 必须标注技术风险等级（🟢低 / 🟡中 / 🔴高），供 Developer 评估。
  - 将设计沉淀至 `Docs/ModuleDesign/...` 对应的 Markdown 文件中。
- **输出/卡点 (Gate)**: 汇报修改摘要，**请求用户 Review**。

## 阶段 2: 代码实现 (@Developer)
- **输入**: 用户通过 `@Developer` 指定的已 Review 过关的设计文档或需求。
- **职责**:
  - **严格遵循**设计文档中的 API 约束和状态定义编写代码。
  - 确保写出的代码支持独立测试（如解耦依赖、提供 Mock 接口或测试场景）。
  - 如果发现设计文档有逻辑漏洞，必须停止编写代码，退回并建议用户呼叫 `@Designer` 修改文档。
  - 对设计文档中标注为 🔴高风险 的项，有权提出替代方案并触发 Design Review。
- **输出/卡点 (Gate)**: 提交完整的代码实现（包含空方法桩或完整逻辑）。汇报代码覆盖的文件，**请求用户 Review**。

## 阶段 3: 测试验证 (@Tester)
- **输入**: 用户通过 `@Tester` 提交的代码和设计文档。
- **职责**:
  - 必须在 `Docs/TestCases/` 目录下建立或更新相关模块的测试文档（例如 `Docs/TestCases/模块名_功能名.md`）。
  - 测试文档必须严格包含两部分：
    1. **测试准备 (Test Preparation)**：列出所有的组件依赖、预制体挂载、Input 绑定与推荐参数等配置前提。
    2. **测试用例 (Test Cases)**：以表格形式提供每个使用场景的具体步骤及对应的**预期表现（Expected Results）**。
  - 编写并执行自动化测试（如果适用，如 PlayMode/EditMode 脚本）。
  - 分析报错堆栈或手感表现。
- **输出/卡点 (Gate)**: 汇报测试准备工作完成，或是提供测试通过率及 Bug 分析报告。如果存在 Bug，建议用户呼叫 `@Developer` 进行修复。
